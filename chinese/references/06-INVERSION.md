# Inversion（需求澄清工作流）

> **作用**：需求不明确时，先收集需求再决定行动
> **适用场景**：需求模糊、目标不清、缺少关键信息
> **加载时机**：决策树判定为「需求不明确」时

---

## 核心原则

**⚠️ 门控：DO NOT start building until all phases are complete**

在需求收集完成之前，禁止开始执行任何实质性工作。

---

## 3 阶段流程

### Phase 1 — 问题发现

**目标**：搞清楚用户真正想解决什么问题

**操作**：
1. 一次只问一个问题
2. 等用户回答后再问下一个
3. 追问关键细节（背景/目的/约束）

**输出**：
```markdown
## 需求理解

### 核心问题
{一句话描述用户想要解决的事}

### 关键细节
- {细节A}
- {细节B}
- {细节C}
```

---

### Phase 2 — 技术约束

**目标**：明确客观限制条件

**操作**：
1. 确认技术栈/平台/工具限制
2. 确认数据/权限/资源限制
3. 确认时间和质量要求

**输出**：
```markdown
## 约束条件

### 技术约束
- {约束A}

### 资源约束
- {约束A}

### 时间/质量约束
- {约束A}
```

---

### Phase 3 — 综合与确认

**目标**：生成完整行动计划，用户确认后再执行

**操作**：
1. 综合 Phase 1 + Phase 2，生成完整计划
2. 用清晰的语言描述：做什么 →怎么做 → 输出什么
3. 等用户确认后再进入执行阶段

**输出**：
```markdown
## 行动计划

### 目标
{一句话目标}

### 执行步骤
1. {步骤1}
2. {步骤2}
3. {步骤3}

### 输出
{交付物描述}

---
请确认以上计划是否符合您的预期，确认后我开始执行。
```

---

## 快速执行清单

```
Inversion 快速执行清单：
□ 加载 inversion.md
□ 发送 Phase 1 第一个问题
□ 等用户回答
□ 追问关键细节（1-3 个问题）
□ 进入 Phase 2：确认约束
□ 进入 Phase 3：生成计划并请求确认
□ 确认后才开始执行
```
